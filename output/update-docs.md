# 文档更新命令：更新实现文档

## 文档分析

1. 检查当前文档状态：
   - 查看 `specs/implementation_status.md` 了解项目整体状态
   - 审查已完成阶段的文档（`specs/phase{N}_implementation_plan.md`）
   - 审查 `specs/flutter_structurizr_implementation_spec.md` 和 `specs/flutter_structurizr_implementation_spec_updated.md`
   - 审查 `specs/testing_plan.md`，确保其根据最近的测试通过、失败和更改保持最新
   - 检查 `CLAUDE.md` 和 `README.md` 了解项目范围内的文档
   - 检查并在 CLAUDE.md 中记录任何新的经验教训或最佳实践

2. 分析实现和测试结果：
   - 审查上一阶段实现的内容
   - 审查测试结果和覆盖率
   - 识别在实现过程中发现的新最佳实践
   - 记录任何实现挑战和解决方案
   - 将更新的文档与最近的实现和测试结果进行交叉引用以确保准确性

## 文档更新

1. 更新阶段实现文档：
   - 使用 ✅ 状态标记已完成的任务
   - 更新实现百分比
   - 添加详细的实现方法说明
   - 记录与原计划的任何偏差并说明理由
   - 根据需要添加新的章节（经验教训、最佳实践）
   - 记录复杂组件的具体实现细节
   - 包含阶段期间发现的任何新故障排除技巧或工作流程改进的摘要

2. 更新实现状态文档：
   - 更新阶段完成百分比
   - 添加或更新组件的实现状态
   - 添加关于实现方法和决策的说明
   - 记录实现过程中发现的最佳实践
   - 记录任何已克服的挑战和已实施的解决方案

3. 更新实现规格文档：
   - 使用 ✅ 或删除线标记已完成的条目，但保留原始需求
   - 在适当的位置添加实现细节说明
   - 添加对已实现文件和类的引用
   - 根据经验更新任何实现指导

4. 必要时更新 CLAUDE.md 和 README.md：
   - 添加新的最佳实践
   - 更新项目状态
   - 添加新的实现指导
   - 记录已知问题或限制
   - 更新使用示例以包含新功能

5. 记录新的测试流程：
   - 添加已创建的测试文件的详细信息
   - 包含测试运行说明
   - 记录测试覆盖率
   - 说明复杂组件的测试方法

## 文档格式与结构

1. 保持一致的文档风格：
   - 使用清晰的标题和章节
   - 在有助于理解的地方包含代码示例
   - 一致使用状态指示符（✅、⚠️、❌）
   - 保持正确的 Markdown 格式

2. 确保文档完整性：
   - 涵盖所有已实现的功能
   - 包含使用示例
   - 记录 API 的更改或新增内容
   - 包含常见问题的故障排除指导

## 指导原则

- 不要创建新的规格文件
- 更新 `specs/` 目录中的现有文件
- 保持一致的文档风格
- 在适当的地方包含实用示例
- 交叉引用相关的文档章节
- 记录最佳实践和经验教训
- 提供清晰的项目进度状态更新
- 更新数字化的完成百分比
- 确保文档反映实际实现情况

完成后提供文档更新摘要，包括：
1. 已更新的文件
2. 文档的主要更改
3. 更新后的完成百分比
4. 已记录的新最佳实践
5. 该阶段后项目的整体状态
